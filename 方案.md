# EcoSort

## 1. 项目要做什么

做一个“能在普通设备上跑得动”的垃圾分类系统：先训练一个准确的基线模型，再把它压缩/加速，并用一个网页 Demo 直观展示 **准确率—速度—体积** 的取舍。

------

## 2. 为什么这个项目值得做

这个项目不只是“训练一个分类器”，而是更像真实工程落地：

- **社会意义**：垃圾分类是现实问题（Social Good）
- **工程意识**：考虑小设备/低功耗场景（System Efficiency）
- **可量化成果**：能用数据说话（准确率、推理时间、模型大小）
- **可展示**：有网页 Demo + 图表 + 视频，比只贴代码更容易让招生官理解

CMU/GT/UMich 都会喜欢这种“有技术、有实验、有落地展示”的作品。

------

## 3. 最终交付物

你最终要交付 4 件东西：

1. **两个模型文件**

- Baseline：未压缩模型（ResNet-18 微调）
- Optimized：压缩后模型（量化或剪枝，至少一种）

1. **一套实验结果**

- 1 张或多张图：压缩强度 vs 准确率/速度/体积
- 1 张对比表：Baseline vs Optimized（数字要清楚）

1. **一个可运行的网页 Demo（Streamlit）**

- 上传图片 → 输出分类结果与置信度
- 同时显示 Baseline 与 Optimized 的推理耗时

1. **一个整理好的 GitHub 仓库**

- README 写清楚动机、方法、结果、复现方式、引用来源
- 代码能复现你展示的结果

------

## 4. 技术路线（用成熟工具）

- 数据集：公开垃圾分类数据（4–6 类即可）
- 模型：ResNet-18（迁移学习）
- 训练：用 GPU 微调（提高效率）
- 压缩/加速（推荐优先做这个）：
  - **量化（PTQ）**：让模型在 CPU 上更快、更小，比较像“边缘设备优化”
  - 可选补充：剪枝（用作对比实验也可以）
- Demo：Streamlit（最省时间，展示效果好）

> 为什么推荐量化：剪枝有时“体积变小但不一定真的加速”，而量化通常能在 CPU 推理上体现更直接的速度收益，更符合“edge”主题。

------

## 5. 15天时间安排（按阶段推进）

### 阶段 A（Day 1–5）：先做出一个准确的 Baseline

目标：先拿到一个“真的能用”的垃圾分类模型（并保存下来）。

主要成果：

- 数据整理好（train/val/test）
- 训练脚本跑通
- Baseline 模型在 test 上有稳定表现
- 有基本评估（accuracy + 混淆矩阵/错误案例）

### 阶段 B（Day 6–9）：做模型压缩并做系统对比实验

目标：得到 Optimized 模型，并用实验说明“压缩到什么程度最划算”。

主要成果：

- 量化模型（或剪枝模型）能正常推理
- 做一组对比实验，输出图表与表格
- 选出最终版本：在精度下降可接受的情况下，速度/体积明显改善

### 阶段 C（Day 10–13）：把结果做成网页 Demo（能给别人看懂）

目标：把技术结果变成直观展示。

主要成果：

- Streamlit 页面：上传图片后，左右对比 Baseline vs Optimized
- 显示：预测类别、置信度、推理耗时、模型大小
- 页面下方展示实验曲线和总结结论

### 阶段 D（Day 14–15）：包装成“申请可用材料”

目标：让它变成能写进申请/面试能讲的项目。

主要成果：

- GitHub README（动机、方法、实验、结果、复现、引用）
- 2 分钟英文 Demo 视频（问题→方法→对比→结论）
